# 玩家模拟系统 Spec

## Why

当前游戏缺乏真实的"游戏内世界模拟"——玩家点击"下一天"时，系统只是简单地增加资源/进度，没有模拟真实玩家群体的行为反馈。玩家作为游戏公司策划，无法感受到"玩家群体"对游戏运营决策的真实反应。我们需要建立一个庞大的"玩家模拟系统"，让每一次运营决策都能获得真实、可信、可感知的反馈。

## 核心设计理念：Agent-Based Simulation

本系统采用**基于代理的模拟（Agent-Based Simulation）**理念：

```
┌─────────────────────────────────────────────────────────────────┐

## What Changes

### 核心架构
- **虚拟玩家池**：预生成10,000个具有真实属性的虚拟玩家
- **玩家画像**：每个玩家有身份、年龄、性格、职业、偏好等Tag
- **分层模拟**：宏观→微观→关键事件，三层架构平衡性能
- **Tag标记驱动**：玩家Tag和内容Tag共同驱动模拟计算
- **AI增强**：部分模拟结果由AI生成，提升真实感

### 系统架构
```
玩家点击"下一天" 
    ↓
SimulationEngine.tick() 
    ↓
┌─────────────────────────────────────────────────────────────────┐
│                     玩家模拟系统核心                              │
│  ┌────────────┐ ┌────────────┐ ┌────────────┐ ┌────────────┐ │
│  │ 宏观计算层 │ │ 微观模拟层  │ │ 关键事件层 │ │ 事件处理层 │ │
│  └────────────┘ └────────────┘ └────────────┘ └────────────┘ │
│                                                                 │
│  模拟内容：                                                     │
│  - 抽卡行为（每个玩家独立决策是否抽卡）                         │
│  - 评论生成（每个玩家独立评价）                                 │
│  - 活动反馈（每个玩家独立参与）                                 │
│  - 社交行为（告白墙、同人广场）                                 │
│  - 留存流失（每个玩家独立状态变化）                            │

    ↓

│                 对玩家公司/项目/角色/剧情的真实影响               │

## ADDED Requirements

### Requirement: 虚拟玩家池系统

系统 SHALL 提供预生成的虚拟玩家池：

#### 玩家画像结构
```typescript
interface VirtualPlayer {
  id: string;
  // 基础属性
  age: number;                    // 年龄：18-45
  gender: '男' | '女' | '其他';
  occupation: string;              // 职业：学生/上班族/自由职业/无业
  
  // 游戏偏好
  genrePreference: string[];       // 题材偏好：甜宠/虐恋/悬疑/治愈
  characterPreference: string[];   // 角色偏好：病娇/温柔/高冷/年下
  plotPreference: string[];       // 剧情偏好：日常/虐心/搞笑/感人
  
  // 消费属性
  spendingLevel: '零氪' | '微氪' | '中氪' | '重氪' | '神豪';
  spendingProbability: number;     // 付费概率：0-1
  gachaMotivation: number;        // 抽卡动机：0-1
  
  // 行为特征
  playStyle: '剧情党' | '强度党' | 'XP党' | '咸鱼党' | '社交党';
  activityLevel: number;          // 活跃度：0-1
  loyalty: number;                // 忠诚度：0-1
  
  // 当前状态
  state: 'NEW' | 'ACTIVE' | 'PAYING' | 'AT_RISK' | 'LOST' | 'RETURNED';
  satisfaction: number;           // 满意度：0-100
  fatigue: number;                // 疲劳度：0-100
  daysSinceLastLogin: number;     // 距上次登录天数
  
  // 历史行为
  history: PlayerHistory;
}
```

#### 玩家池生成策略
- **总量**：10,000个虚拟玩家
- **分布**：
  - 年龄：18-25岁40%，26-35岁45%，36-45岁15%
  - 性别：女80%，男18%，其他2%
  - 职业：学生25%，上班族50%，自由职业15%，无业10%
  - 付费：零氪50%，微氪30%，中氪15%，重氪4%，神豪1%

#### Scenario: 玩家池初始化
- **GIVEN** 系统需要初始化玩家池
- **WHEN** 初始化完成
- **THEN** 生成10,000个虚拟玩家
- **AND** 每个玩家有完整的画像属性
- **AND** 初始状态为NEW，满意度70，疲劳度10

### Requirement: 分层模拟架构

系统 SHALL 通过三层架构平衡性能和真实性：

#### Layer 1: 宏观计算（毫秒级）
- **目的**：快速估算整体数据
- **输入**：当前项目属性、活动配置、赛季状态
- **输出**：DAU、留存、收入、满意度的预估值
- **算法**：基于Tag的影响矩阵 + 历史数据回归

```typescript
interface MacroResult {
  estimatedDAU: number;
  estimatedRetention: { d1: number; d7: number; d30: number };
  estimatedRevenue: number;
  estimatedSatisfaction: number;
  confidence: number;  // 置信度
}
```

#### Layer 2: 微观模拟（100-500玩家）
- **目的**：模拟具体玩家行为，获得真实分布
- **输入**：宏观结果 + 采样玩家
- **输出**：每个玩家的行为记录 + 统计分布
- **算法**：每个玩家独立决策

```typescript
interface MicroSimulation {
  sampledPlayers: VirtualPlayer[];  // 采样的玩家
  behaviors: PlayerBehavior[];      // 行为记录
  statistics: {
    gachaDistribution: Record<string, number>;
    activityParticipation: Record<string, number>;
    sentimentDistribution: { positive: number; neutral: number; negative: number };
  };
}
```

#### Layer 3: 关键事件（10-50玩家）
- **目的**：深度模拟极端情况和舆情事件
- **触发条件**：
  - 满意度<30%时高概率触发
  - 随机1%概率触发
  - 特定Tag组合触发
- **输出**：极端行为详情、舆情分析

#### 执行流程
```
tick() 
  → Layer1: 宏观计算（5ms）
      ↓
  Layer2: 微观模拟（50-200ms，200玩家）
      ↓
  Layer3: 关键事件（如需要，10ms）
      ↓
  汇总结果 → 应用到数据
```

### Requirement: 抽卡行为模拟

系统 SHALL 模拟每个玩家的抽卡决策：

#### 决策流程
```
玩家是否抽卡？
  ├── 基础概率 = 玩家.抽卡动机 × 活动.吸引力
  ├── Tag匹配 = 玩家.角色偏好 ∩ UP角色.Tag
  ├── 保底影响 = 距上次SSR天数 × 0.1
  └── 最终概率 = 基础 + Tag加成 + 保底影响
```

#### 抽卡结果
- **概率**：SSR 3%, SR 20%, R 77%（可配置）
- **UP影响**：UP角色概率×1.5
- **保底**：90抽必出SSR

#### Tag影响示例
```
玩家偏好"病娇" + UP角色有"病娇"Tag
  → 抽卡概率 +30%
  
玩家是"强度党" + 角色强度高
  → 抽卡概率 +20%
  
玩家是"XP党" + 角色立绘好看
  → 抽卡概率 +15%
```

#### Scenario: 玩家抽卡模拟
- **GIVEN** 玩家A：甜宠控、微氪、强度党；UP角色"冰山总裁"Tag:高冷、温柔
- **WHEN** 模拟运行
- **THEN** 抽卡概率 = 0.3(基础) + 0(无匹配) + 0(无保底) = 30%
- **AND** 玩家决定抽卡
- **AND** 抽取结果按概率分布
- **AND** 收入 = 18钻石 × 抽卡次数

### Requirement: 评论生成模拟

系统 SHALL 模拟每个玩家的评论行为：

#### 决策流程
```
玩家是否评论？
  ├── 活跃度门槛：activityLevel > 0.3
  ├── 满意度影响：满意度>70 +20%，<30 -30%
  └── 事件触发：好评/差评事件后概率提升
```

#### 评论平台分配
- **TapTap**：剧情党 + 活跃用户
- **应用商店**：所有用户
- **微博**：社交党 + 话题敏感
- **B站**：视频爱好者 + XP党
- **小红书**：女性用户 + 颜值党

#### Tag影响内容
```
玩家.剧情偏好："虐恋" + 角色.有"病娇"Tag
  → 评论内容："病娇太香了，虐恋yyds"
  
玩家.满意度：85 + 活动.高福利
  → 评论内容："官方大气，这次活动太划算了"
  
玩家.疲劳度：80
  → 评论内容："玩法太肝了，长草期太长"
```

#### Scenario: 评论生成
- **GIVEN** 玩家B：剧情党、满意度85、微博偏好
- **WHEN** 模拟运行
- **AND** 触发评论生成
- **THEN** 生成微博评论
- **AND** 内容关联剧情Tag
- **AND** 情感正面

### Requirement: 活动参与模拟

系统 SHALL 模拟每个玩家的活动参与：

#### 决策因素
```
参与概率 = 基础参与率 
  × 活动吸引力(奖励价值/难度)

```
活动Tag："情人节" + 玩家.浪漫属性
  → 参与率 +30%
  
活动Tag："高难度" + 玩家.是"咸鱼党"
  → 参与率 -50%
  
活动奖励丰富 + 玩家.是"微氪"
  → 参与率 +20%
```

### Requirement: 玩家状态模拟

系统 SHALL 模拟每个玩家的状态变化：

#### 状态机
```
NEW ──(D1留存)──→ ACTIVE ──(付费)──→ PAYING
                      ↑               │
                      │             流失
                   回归              ↓
                  RETURNED ←──── AT_RISK ──→ LOST
```

#### 状态转换规则
```
AT_RISK判定条件（满足任一）：
  - 满意度 < 40
  - 疲劳度 > 80
  - 连续7天未登录
  
LOST判定条件（满足任一）：
  - 满意度 < 20
  - 连续14天未登录
  - 疲劳度 > 95
  
RETURNED触发条件：
  - 流失玩家被召回活动吸引
  - 概率 = 召回活动吸引力 × 0.1
```

#### Tag影响留存
```
项目Tag.甜宠 → D7留存 +15%
项目Tag.虐恋 → D30留存 +10%，但D1 -10%
角色Tag.病娇 → 核心玩家留存 +20%，新手 -15%
剧情.长 → 在线时长 +30%，流失风险 +10%
```

### Requirement: 社交行为模拟

系统 SHALL 模拟玩家的社交行为：

#### 告白墙行为
```
发言概率 = 活跃度 × 情感表达欲 × 事件触发
类型分布：

```
创作概率 = 核心玩家标识 × 创作欲
类型分布：

  - 文稿：35%（角色.CP热度 > 阈值）
  - 视频：15%（剧情.适合视频化）

```

### Requirement: 全局影响计算

系统 SHALL 汇总所有玩家行为计算对游戏数据的影响：

#### 对公司的影响

| 影响项 | 计算方式 | 示例 |
|--------|----------|------|
| **收入** | 抽卡收入 + 活动收入 + 付费收入 - 退款 | 抽卡500次×18钻=9000钻 |
| **支出** | 运营成本 + 福利成本 + 赔偿成本 | 活动奖励1000钻+补偿500钻=1500钻 |
| **现金流** | 上期现金流 + 本期收入 - 本期支出 | 10000 + 9000 - 1500 = 17500 |
| **声誉** | 口碑变化值累加 | 好评+5，差评-3，净+2 |
| **品牌价值** | 长期积累，按月计算 | 持续好评品牌+1%/月 |

#### 对项目的影响

| 影响项 | 计算方式 | 来源 |
|--------|----------|------|
| **DAU** | 活跃玩家数量统计 | 玩家状态模拟 |
| **新增用户** | NEW状态玩家数 | 玩家状态模拟 |
| **留存率D1** | 次日回访/N新增 | 玩家状态模拟 |
| **留存率D7** | 7日后回访/N新增 | 玩家状态模拟 |
| **留存率D30** | 30日后回访/N新增 | 玩家状态模拟 |
| **付费率** | 付费玩家/活跃玩家 | 玩家状态模拟 |
| **ARPU** | 总收入/活跃玩家 | 收入/DAU |
| **综合评分** | 各平台评分加权平均 | 五平台评价 |
| **满意度** | 玩家满意度平均值 | 玩家状态 |
| **疲劳度** | 玩家疲劳度平均值 | 玩家状态 |

#### 对角色的影响

| 影响项 | 计算方式 | 来源 |
|--------|----------|------|
| **人气值** | 抽取次数×2 + 评论提及×1 + 同人×3 | 抽卡+评论+同人 |
| **抽取率** | 该角色抽取次数/总抽取 | 抽卡模拟 |
| **讨论度** | 各平台提及次数之和 | 评论模拟 |
| **CP热度** | 与其他角色组合出现次数 | 评论+同人 |
| **好评率** | 正面评论/总评论 | 评论模拟 |
| **流失贡献** | 因该角色离开的玩家数 | 玩家状态 |

**示例**：
```
角色"冰山总裁"：
  - 人气值 = 抽取200次×2 + 讨论50次×1 + 同人20件×3 = 560
  - 抽取率 = 200/1000 = 20%
  - CP热度（×温柔） = 80
  - 好评率 = 45/50 = 90%
```

#### 对剧情的影响

| 影响项 | 计算方式 | 来源 |
|--------|----------|------|
| **完成率** | 完成人数/开始人数 | 玩家行为 |
| **好评率** | 正面评价/总评价 | 评论模拟 |
| **流失点** | 大量玩家放弃的章节 | 玩家行为 |
| **热门选项** | 选择最多的分支 | 玩家行为 |
| **平均时长** | 完成剧情的平均时间 | 玩家行为 |
| **重玩率** | 重新开始该剧情的玩家比例 | 玩家行为 |

**示例**：
```
剧情"情人节特别篇"：
  - 完成率 = 800/1000 = 80%
  - 好评率 = 720/800 = 90%
  - 流失点 = 第3章(150人放弃)、第5章(80人放弃)
  - 热门选项 = A选项(60%)、B选项(30%)、C选项(10%)
```

### Requirement: 影响传导机制

系统 SHALL 实现玩家行为→子模块→公司/项目/角色/剧情的完整传导：

#### 传导链路

```
玩家行为
    │
    ├──→ 抽卡 ──────────────→ 角色.人气↑ + 角色.抽取率↑ + 项目.收入↑ + 公司.收入↑
    │
    ├──→ 评论 ──────────────→ 项目.评分↑/↓ + 项目.满意度↑/↓ + 角色.讨论度↑
    │
    ├──→ 活动参与 ──────────→ 项目.DAU↑ + 项目.满意度↑ + 公司.收入↑
    │
    ├──→ 社交（告白）───────→ 项目.满意度↑ + 角色.人气↑
    │
    ├──→ 社交（同人）───────→ 角色.讨论度↑ + 角色.CP热度↑ + 项目.口碑↑
    │
    └──→ 状态变化 ──────────→ 项目.DAU↑/↓ + 项目.留存↑/↓ + 项目.付费率↑/↓
                                    │
                                    └──→ 公司.收入↑/↓
```

## 技术实现要点

### 核心类设计
```typescript
class SimulationEngine {
  private playerPool: VirtualPlayerPool;
  private config: SimulationConfig;
  
  async tick(): Promise<SimulationResult> {
    // Layer 1: 宏观计算
    const macro = this.macroLayer.calculate();
    
    // Layer 2: 微观模拟
    const sampled = this.playerPool.sample(200);
    const micro = await this.microLayer.simulate(sampled);
    
    // Layer 3: 关键事件
    const critical = this.criticalLayer.check(micro);
    
    // 汇总结果
    return this.aggregate(macro, micro, critical);
  }
}

class VirtualPlayerPool {
  private players: VirtualPlayer[];
  
  initialize(): void;  // 生成10,000玩家
  sample(count: number): VirtualPlayer[];  // 随机采样
  getByTag(tags: string[]): VirtualPlayer[];  // 按Tag筛选
}
```
