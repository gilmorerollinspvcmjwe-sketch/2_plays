# 乙游模拟器核心玩法重构 Spec

## Why

当前游戏的首页流程分散且缺少"创建项目"的核心步骤，运营管理的7大模块之间缺乏相互影响机制，玩家无法通过运营数据有效改进游戏。我们需要重构核心玩法循环，建立真实的模拟系统，让乙女玩家作为游戏公司老板或策划，真实感受到"理想与现实"的反差感——想要打造完美乙游，却受限于资源、团队能力、市场反馈等多重因素。

## What Changes

### 核心流程重构
- **新增"项目"概念**：游戏以项目为单位进行管理，一个公司可同时开发/运营多个项目
- **标准化流程**：创建公司 → 创建项目 → 组建团队 → 开发角色 → 编写剧情 → 开发游戏 → 发布游戏 → 持续迭代反馈
- **项目生命周期**：每个项目有完整的生命周期管理，从立项到停服

### 运营模块联动系统
- **7大运营模块相互影响**：卡池/活动/事件/玩家数据/福利/社交/市场之间存在真实的因果关系链
- **数据反馈闭环**：运营数据 → 影响计算 → 反馈到项目/角色/剧情 → 玩家决策调整

### 计算系统架构
- **项目评估系统**：实时计算项目健康度、收益预期、玩家满意度
- **角色价值系统**：计算角色人气、CP价值、剧情承载力
- **剧情分析系统**：分析剧情完成率、玩家流失点、情感曲线效果
- **运营影响系统**：计算运营决策对各项指标的短期/长期影响

### 长期玩法设计
- **赛季制**：每季度市场趋势变化，需要调整运营策略
- **团队成长**：团队成员有技能成长、疲劳度、离职风险
- **竞争环境**：竞品游戏动态影响市场份额
- **玩家社群演化**：核心玩家、意见领袖的形成与流失

## Impact

- **受影响的核心文件**：
  - `src/stores/gameStore.ts` - 项目管理体系重构
  - `src/stores/operationStore.ts` - 运营联动机制
  - `src/views/HomeView.vue` - 首页流程重新设计
  - `src/views/creator/` - 创建流程整合到项目体系
  - 新增计算系统模块

- **玩家体验变化**：
  - 从"零散创建"到"项目化管理"
  - 从"单点运营"到"系统联动"
  - 从"静态反馈"到"动态演化"

## ADDED Requirements

### Requirement: 项目管理体系

系统 SHALL 建立以项目为核心的管理体系：

#### 项目生命周期
- **立项阶段**：确定项目定位、目标受众、预算范围
- **开发阶段**：角色开发、剧情编写、技术实现并行进行
- **发布阶段**：正式上线、开服活动
- **运营阶段**：日常运营、版本更新、活动策划、持续迭代
- **衰退/停服阶段**：玩家流失管理、项目收尾

#### 项目属性
- 项目名称、定位（大众/小众）、题材标签
- 预算池（开发预算+运营预算）
- 开发进度（0-100%）
- 项目健康度（综合评分）
- 团队配置（策划/美术/程序/运营人数）

#### 项目间关系
- 资源共享：同一公司多个项目共享部分资源
- 团队调度：人员可在项目间流动
- 品牌效应：成功项目带动新项目
- 竞争关系：同公司项目可能互相抢用户

#### Scenario: 创建新项目
- **WHEN** 玩家在首页点击"创建新项目"
- **THEN** 进入项目立项流程：选择定位 → 确定预算 → 组建团队 → 设定里程碑
- **AND** 项目创建后可在首页看到项目卡片，显示当前阶段和进度

#### 项目卡片信息
- 项目封面（可自定义）
- 项目名称 + 当前阶段标签
- 进度条（开发进度/运营天数）
- 关键指标预览（DAU/收入/评分）
- 待处理事项红点
- 点击进入项目详情

#### 全局数据看板
- 公司总收入/总支出/净利润
- 活跃项目数/总项目数
- 团队总人数/平均疲劳度
- 市场份额占比

#### Scenario: 首页项目管理
- **WHEN** 玩家打开首页
- **THEN** 看到所有项目的整体状态
- **AND** 可以快速跳转到需要处理的项目
- **AND** 通过数据看板了解公司整体经营状况

### Requirement: 员工/团队系统

系统 SHALL 建立完整的员工管理系统：

#### 员工属性
- 基础信息：姓名、职位（策划/美术/程序/运营）、等级（初级/中级/高级/专家）
- 技能属性：各岗位专业技能值（0-100）
- 状态属性：疲劳度（0-100）、满意度（0-100）、薪资期望
- 特殊属性：专长领域、性格特质

#### 招聘系统
- 发布招聘信息（选择岗位、薪资范围、技能要求）
- 简历筛选（查看应聘者资料）
- 面试评估（技能测试 + 性格匹配）
- 录用入职

#### 员工成长
- 项目经验积累（参与项目获得经验）
- 技能提升（培训、实践提升技能值）
- 等级晋升（满足条件后晋升，薪资相应提升）
- 专长培养（培养特定领域的专家）

#### 员工管理
- 分配项目（将员工分配到不同项目）
- 调整薪资（影响满意度）
- 加班安排（提升进度但增加疲劳）
- 离职处理（疲劳过高或满意度低会离职）

#### Scenario: 团队组建
- **WHEN** 创建新项目时
- **THEN** 需要为项目配置团队（策划x2、美术x3、程序x2、运营x1）
- **AND** 可以从现有员工中调配，也可以招聘新员工
- **AND** 团队配置影响项目开发速度和质量

### Requirement: 运营模块联动系统

系统 SHALL 建立7大运营模块之间的真实影响机制：

#### 模块关系图
```
                    ┌─────────────┐
                    │   市场情报   │
                    └──────┬──────┘
                           │ 影响趋势
                           ▼
┌─────────┐    ┌─────────────┐    ┌─────────┐
│  卡池   │◄──►│   活动      │◄──►│  福利   │
│  管理   │    │   中心      │    │  发放   │
└────┬────┘    └──────┬──────┘    └────┬────┘
     │                │                │
     │ 影响收入       │ 影响热度       │ 影响满意度
     ▼                ▼                ▼
┌─────────────┐    ┌─────────────┐    ┌─────────┐
│  玩家数据   │◄──►│  运营事件   │◄──►│  社交   │
│  （核心）   │    │  （触发器）  │    │  广场   │
└─────────────┘    └─────────────┘    └─────────┘
```

#### 联动规则示例

**卡池 → 玩家数据**：
- 爆率设置影响玩家满意度
- UP角色选择影响不同玩家群体
- 卡池频率影响玩家付费意愿

**活动 → 运营事件**：
- 活动福利好 → 可能触发"福利节奏"（玩家要求更多）
- 活动难度高 → 可能触发"太肝了"事件
- 活动奖励差 → 可能触发"抠门官方"事件

**玩家数据 → 市场情报**：
- DAU下降 → 市场份额降低
- 评分下降 → 口碑传播负面
- 付费率变化 → 影响收入预期

**运营事件 → 社交广场**：
- 爆率争议事件 → 社交广场出现大量吐槽
- 剧情好评 → 同人创作增加
- 危机处理不当 → 告白墙出现退坑告白

#### 反馈延迟机制
- **即时反馈**：福利发放、卡池调整等立即看到效果
- **短期反馈（1-3天）**：活动效果、小范围口碑变化
- **中期反馈（1-2周）**：玩家留存变化、收入趋势
- **长期反馈（1月+）**：品牌影响力、市场份额变化

#### Scenario: 运营决策连锁反应
- **WHEN** 玩家决定降低卡池爆率从2%到1%
- **THEN** 3天内：评论区出现大量吐槽，满意度下降
- **AND** 1周内：部分玩家流失，DAU下降
- **AND** 2周内：收入短期上升但口碑恶化
- **AND** 1月后：市场份额被竞品抢占
- **FINALLY** 玩家需要发福利/提高爆率来挽回

### Requirement: 计算系统架构

系统 SHALL 建立四个核心计算系统，支撑模拟的真实性：

#### 1. 项目评估系统

**输入**：
- 角色完成度、质量评分
- 剧情完成度、分支丰富度
- 技术实现度（Bug率、性能）
- 团队配置合理性

**输出**：
- 项目健康度（0-100）
- 预期上线评分（1-10）
- 预期首月收入
- 风险评估（高/中/低）

**计算公式示例**：
```
项目健康度 = 角色完成度×0.3 + 剧情完成度×0.3 + 技术实现度×0.2 + 团队匹配度×0.2

预期评分 = 基础分(5) + 角色质量×0.5 + 剧情质量×0.5 + 技术分×0.3 - Bug扣分

预期收入 = 市场容量 × 预期评分/10 × 营销系数 × 题材热度系数
```

#### 2. 角色价值系统

**输入**：
- 角色属性（外貌、性格、声优等）
- 剧情参与度
- 玩家互动数据（亲密度、约会次数）
- 卡池表现（抽取率、保有量）

**输出**：
- 角色人气值（排名）
- 角色商业价值（带货能力）
- CP价值（与其他角色的化学反应）
- 剧情承载力（适合什么类型的剧情）

**计算公式示例**：
```
角色人气 = 基础人气 + 剧情曝光×0.3 + 卡池抽取×0.4 + 玩家互动×0.3

CP热度 = 两角色人气之和 × 互动剧情质量 × 玩家投票

商业价值 = 人气值 × 付费转化率 × 周边开发潜力
```

#### 3. 剧情分析系统

**输入**：
- 剧情结构（章节数、分支数、结局数）
- 玩家行为数据（完成率、选择分布、重玩率）
- 评论情感分析
- 流失点分析

**输出**：
- 剧情完成率（多少玩家看到结局）
- 情感曲线效果（是否达到预期情绪）
- 关键流失点（哪里玩家放弃）
- 热门分支（玩家最喜欢的选择）

**计算公式示例**：
```
完成率 = 到达结局玩家数 / 开始剧情玩家数

情感效果 = Σ(预期情绪值 - 实际情绪值)² / 数据点数量

流失风险点 = 玩家流失率突增的章节/选项
```

#### 4. 运营影响预测系统

**输入**：
- 运营决策（爆率、福利、活动内容）
- 当前玩家状态（满意度、活跃度、付费意愿）
- 市场环境（竞品动态、趋势变化）
- 历史数据（同期表现、环比变化）

**输出**：
- 短期影响预测（1-7天）
- 中期影响预测（1-4周）
- 长期影响预测（1-3月）
- 风险预警（可能出现的问题）

**计算公式示例**：
```
满意度变化 = 决策满意度系数 × 玩家群体敏感度 × 时间衰减

收入预测 = 当前收入 × (1 + 运营系数 + 市场系数 + 随机波动)

流失风险 = 基础流失率 + 不满系数×0.5 + 竞品吸引×0.3 + 疲劳度×0.2
```

#### Scenario: 计算系统支撑决策
- **WHEN** 玩家准备设置新卡池爆率
- **THEN** 系统显示预测：
  - "当前设置1%爆率，预计："
  - "3天内满意度下降15%，可能出现吐槽节奏"
  - "1周内DAU下降8%，但收入上升20%"
  - "建议搭配福利活动平衡玩家情绪"
- **AND** 玩家可以调整参数实时看到预测变化

### Requirement: 数据反馈与改进闭环

系统 SHALL 建立从运营数据到游戏改进的完整闭环：

#### 数据收集层
- **玩家行为数据**：登录、游玩、付费、流失
- **内容消费数据**：剧情完成率、角色偏好、卡池抽取
- **社交反馈数据**：评论情感、热点话题、同人创作
- **市场数据**：竞品动态、趋势变化、市场份额

#### 数据分析层
- **自动分析报告**：每日/每周自动生成运营报告
- **异常检测**：自动发现数据异常（突然流失、收入暴跌）
- **归因分析**：分析问题原因（是爆率问题？还是剧情问题？）
- **趋势预测**：预测未来7天/30天走势

#### 决策建议层
- **问题诊断**："玩家流失主要原因是第三章剧情过于虐心"
- **改进建议**："建议增加甜宠剧情分支平衡情绪"
- **效果预测**："预计修改后完成率提升20%，满意度提升15%"
- **成本估算**："修改需要3天开发时间，消耗500开发点"

#### 执行反馈层
- **改进执行**：玩家根据建议调整游戏内容
- **效果追踪**：实时追踪改进后的数据变化
- **ROI计算**：计算改进投入产出比
- **经验沉淀**：成功的改进策略形成知识库

#### Scenario: 数据驱动的游戏优化
- **GIVEN** 项目A上线后数据显示第三章完成率仅30%
- **WHEN** 系统分析发现该章节虐心剧情过多且无分支
- **THEN** 系统建议："增加甜宠分支选项，预计完成率提升至50%"
- **AND** 玩家执行修改
- **THEN** 一周后数据显示完成率提升至48%，满意度上升
- **AND** 系统记录："甜宠分支对虐恋剧情有显著平衡作用"

### Requirement: 长期玩法机制

系统 SHALL 设计支撑长期游玩的机制：

#### 赛季制市场变化
- **每季度**：市场趋势变化（甜宠/虐恋/悬疑题材热度轮换）
- **每半年**：大型行业事件（游戏展、年度评选）
- **每年**：技术革新（新渲染技术、新交互方式）

#### 团队成长系统
- **技能成长**：团队成员随项目经验提升技能
- **疲劳管理**：长期加班导致效率下降、离职风险
- **招聘培养**：从新人培养到资深，或高薪挖角
- **团队文化**：不同团队组合产生不同化学反应

#### 竞争环境演化
- **竞品动态**：其他虚拟游戏公司发布新作
- **市场洗牌**：表现不佳的游戏被淘汰
- **合作机会**：与其他公司联动、IP合作
- **行业事件**：政策变化、技术突破、社会热点

#### 玩家社群演化
- **核心玩家**：长期活跃玩家形成核心社群
- **意见领袖**：产生有影响力的玩家KOL
- **玩家分化**：不同偏好的玩家群体形成
- **社群事件**：玩家自发组织的活动、争议

#### Scenario: 长期经营的挑战
- **GIVEN** 玩家成功运营项目A一年，成为头部乙游
- **WHEN** 新赛季开始，市场趋势转向悬疑题材
- **AND** 团队核心策划被竞品挖角
- **AND** 核心玩家群体对现状不满，要求创新
- **THEN** 玩家需要：
  - 决定是坚持甜宠路线还是转型悬疑
  - 紧急招聘或培养新策划
  - 平衡老玩家需求与新玩家吸引
  - 可能启动新项目B探索新题材

## MODIFIED Requirements

### Requirement: 角色创建流程
**修改内容**：角色创建必须在项目框架内进行，角色归属特定项目
**原因**：建立项目-角色的从属关系，支持跨项目角色复用和对比

### Requirement: 剧情设计流程
**修改内容**：剧情设计绑定到具体项目和角色，支持项目内剧情管理
**原因**：明确内容归属，便于项目级数据分析和改进

### Requirement: 运营管理系统
**修改内容**：运营操作以项目为单位，增加模块间联动计算
**原因**：实现真实的运营模拟，让玩家感受决策的连锁反应

## REMOVED Requirements

### Requirement: 项目测试阶段
**移除内容**：原流程中的"测试阶段"已移除
**原因**：简化流程，将测试融入开发阶段，更符合实际游戏开发流程

## 玩家体验目标

### 核心反差感
1. **理想与现实的反差**：想做好游戏，但受限于资源、团队、市场
2. **决策与后果的反差**：短期利益与长期口碑的权衡
3. **投入与回报的反差**：精心打磨的内容可能不如跟风题材

### 成长曲线
- **新手期（1-2周）**：学习基础流程，第一个项目可能失败
- **成长期（1-2月）**：掌握运营技巧，项目开始盈利
- **成熟期（3-6月）**：多项目管理，应对复杂市场变化
- **大师期（6月+）**：建立游戏帝国，培养新人制作人

### 情感体验
- **焦虑**：资源不足、团队离职、竞品威胁
- **成就感**：项目上线、口碑爆棚、收入破纪录
- **纠结**：道德抉择（逼氪vs口碑、赶工vs质量）
- **惊喜**：黑马项目、意外爆款、玩家二创
