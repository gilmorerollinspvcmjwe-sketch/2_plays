## Task 5: 运营模块联动系统 - 基础联动

### 目标
建立7大运营模块（卡池/活动/事件/玩家数据/福利/社交/市场）之间的真实影响机制，让玩家感受到运营决策的连锁反应。

### 核心设计

#### 1. 模块关系架构

```
                    ┌─────────────┐
                    │   市场情报   │
                    └──────┬──────┘
                           │ 影响趋势
                           ▼
┌─────────┐    ┌─────────────┐    ┌─────────┐
│  卡池   │◄──►│   活动      │◄──►│  福利   │
│  管理   │    │   中心      │    │  发放   │
└────┬────┘    └──────┬──────┘    └────┬────┘
     │                │                │
     │ 影响收入       │ 影响热度       │ 影响满意度
     ▼                ▼                ▼
┌─────────────┐    ┌─────────────┐    ┌─────────┐
│  玩家数据   │◄──►│  运营事件   │◄──►│  社交   │
│  （核心）   │    │  （触发器）  │    │  广场   │
└─────────────┘    └─────────────┘    └─────────┘
```

#### 2. 联动规则库设计

**规则数据结构：**
```typescript
interface LinkageRule {
  id: string;
  sourceModule: OperationModule;
  targetModule: OperationModule;
  condition: RuleCondition;
  effect: RuleEffect;
  delay: FeedbackDelay;
}

type OperationModule = 
  | 'gacha'      // 卡池
  | 'event'      // 活动
  | 'welfare'    // 福利
  | 'playerData' // 玩家数据
  | 'incident'   // 运营事件
  | 'social'     // 社交广场
  | 'market';    // 市场情报

interface RuleCondition {
  type: 'threshold' | 'change' | 'comparison';
  metric: string;
  operator: '>' | '<' | '==' | '>=' | '<=';
  value: number;
}

interface RuleEffect {
  targetMetric: string;
  changeType: 'absolute' | 'percentage';
  changeValue: number;
  duration: number; // 持续天数
}

type FeedbackDelay = 'immediate' | 'short' | 'medium' | 'long';
```

**核心联动规则：**

| 源模块 | 目标模块 | 触发条件 | 影响效果 | 延迟 |
|--------|----------|----------|----------|------|
| 卡池 | 玩家数据 | 爆率 < 2% | 满意度 -15% | 即时 |
| 卡池 | 玩家数据 | UP角色人气高 | 付费率 +20% | 即时 |
| 活动 | 运营事件 | 福利价值 > 100 | 触发"福利节奏" | 短期 |
| 活动 | 运营事件 | 活动难度 > 80 | 触发"太肝了" | 短期 |
| 福利 | 玩家数据 | 发放钻石 > 1000 | 满意度 +10% | 即时 |
| 玩家数据 | 市场情报 | DAU下降 > 10% | 市场份额 -5% | 中期 |
| 运营事件 | 社交广场 | 负面事件 | 吐槽帖子 +50% | 即时 |
| 市场情报 | 卡池 | 竞品活动 | 收入 -15% | 中期 |

## Task 6: 角色价值系统

### 目标
实现角色人气、商业价值、CP价值的计算，帮助玩家了解哪些角色受欢迎，如何优化角色设计。

### 核心设计

#### 1. 角色价值数据结构

#### 2. 计算公式

**角色人气：**
```
人气值 = 基础分(30) 
       + 剧情曝光度×0.25 
       + 卡池抽取量×0.35 
       + 社交提及度×0.20 
       + 玩家互动×0.20

剧情曝光度 = 该角色出场章节数 / 总章节数 × 100
卡池抽取量 = 该角色被抽取次数 / 总抽取次数 × 100
社交提及度 = 该角色被提及次数 / 总提及次数 × 100
玩家互动 = 约会次数 + 礼物赠送 + 亲密度增长
```

**商业价值：**
```
商业价值 = 人气值×0.4 
         + 付费转化率×0.3 
         + 周边开发潜力×0.3

付费转化率 = 该角色卡池付费人数 / 看到卡池人数
周边开发潜力 = 基于角色设计复杂度、辨识度评估
```

**CP热度：**
```
CP热度 = (角色A人气 + 角色B人气) / 2 
       × 互动剧情质量 
       × 玩家投票支持率

互动剧情质量 = 玩家选择该CP比例 × 剧情评分
```

### 目标

```typescript
interface ChapterCompletion {

  chapterTitle: string;

}

interface EmotionPoint {

```

#### 2. 计算公式

**完成率：**
```
章节完成率 = 完成该章节人数 / 开始该章节人数 × 100%
总体完成率 = 看到结局人数 / 开始剧情人数 × 100%

流失率 = (开始人数 - 完成人数) / 开始人数 × 100%
```

**情感曲线偏离度：**
```
偏离度 = Σ|预期情绪值 - 实际情绪值| / 数据点数量

效果评分 = 100 - 偏离度
```

**分支热度：**
```
分支选择比例 = 选择该分支人数 / 总选择人数 × 100%

热门分支 = 选择比例 > 40% 的分支
冷门分支 = 选择比例 < 5% 的分支
```

#### 3. 流失点识别算法

```typescript
function identifyDropOffPoints(
  chapterData: ChapterCompletion[]
): DropOffPoint[] {
  const dropOffs: DropOffPoint[] = [];
  
  for (let i = 0; i < chapterData.length - 1; i++) {
    const current = chapterData[i];
    const next = chapterData[i + 1];
    
    const dropOffRate = (current.completeCount - next.startCount) / current.completeCount;
    
    if (dropOffRate > 0.2) { // 流失率超过20%
      dropOffs.push({
        location: `${current.chapterTitle} -> ${next.chapterTitle}`,
        dropOffRate,
        possibleReasons: analyzeDropOffReason(current, next)
      });
    }
  }
  
  return dropOffs;
}

function analyzeDropOffReason(
  current: ChapterCompletion,
  next: ChapterCompletion
): string[] {
  const reasons: string[] = [];

    reasons.push('本章完成率过低，内容可能过于困难或枯燥');
  }
  
  // 基于章节内容分析其他可能原因
  // ...
  
  return reasons;
}
```

## 技术实现要点

### 1. 数据流设计

```
玩家操作 → 数据收集 → 实时分析 → 问题诊断 → 建议生成 → 报告展示
                ↓
         历史数据存储 ← 趋势分析
```

### 2. 性能优化

- **计算缓存**：复杂计算结果缓存，避免重复计算
- **增量更新**：只更新变化的数据
- **异步处理**：非关键计算异步执行
- **数据分页**：大量数据分页加载
